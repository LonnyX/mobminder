OBJECTIFS:


o Pas d'auto-recover coté SPARE, car en cas de hacking on veut pouvoir faire un assess de la qualité de la DB, avant de l'utiliser.
o On souhaite produire des backups réguliers toutes les 10 minutes.


---------------------------------------------------------------------------------------------------

PRINCIPES de mise en oeuvre:

o Le serveur prod a deux serveurs mysql: mysql (ouvert en écriture pour php) et mysql-slave
	user pour mysql-slave: mysql-slave

o La prod produit un backup de mysql-slave/mobminder toutes les 10min, gzippé .tar
	( en cas de hacking, on veut pouvoir retourner X * 10min en arrière )

o Le spare scp par ssh le .tar depuis PRODUCTION vers SPARE 

On souhaite garder:
	o 1 backup toute les 10min pendant 2h
	o 1 backup toutes heure pendant 24h

	Evaluation du poids:
	100M = 1 backup zippé
	3 x 6 x 500M = 1.8G
	48 x 500M = 4.8G
	Total = 25G  => on réserve 50G sur app02 ( total 12+50 = 64G sur )

Mise en oeuvre bash:

Coté SPARE: user: fetch-slave

	On a deux bash commandés par crons: 
	 A- tens.sh à l'heure 09,19,29,39,49,59
	 B- hourly.sh à l'heure 08

	Scénario des bash:
	step 1: scp le .gz depuis la prod, dans un dossier /tens respectivement /hourly ( utilisateur pour le scp: dump-slave )
	step 2: supprimer le .gz sur la prod (uniquement le cron /tens.sh)
	step 3: effacer dans /tens les fichiers plus anciens que now-3h
			dans /hourly les fichiers plus anciens que now-48h


Coté PROD: user: dump-slave

	On a un bash commandé par cron:
	C- spare.sh à l'heure 02 (, 12, 22, 32, 42, 52, temporairement une fois par heure )

	Scénario du bash /srv/syst/dump-slave.sh:

	step1: Lock la DB mysql-slave/mobminder
	step2: mySQLdump vers /tmp/slavebackup
	step3: gzip le dump. 
	step4: mvle .gz vers /home/mysql-slave/yyyy-mm-dd_hh:mm.gz
	step5: mv le gz dans un dossier accessible pour le spare (qui va se logger comme mysql-slave)


---------------------------------------------------------------------------------------------------

IMPLEMENTATION: 


- SRV PRODUCTION:
	une cron exécute régulièrement le scripte  /srv/syst/dump-slave.sh
	la commande de cron est le fichier /etc/cron.d/dump-slave.sh


- SRV SPARE, les backups se trouvent dans:


/home/fetch-slave/tens/ (pour les 3 dernières heures)

et

/home/fetch-slave/hourly/ (pour les 48 dernières heures)


---------------------------------------------------------------------------------------------------

PROCEDURE de BACKUP RESTORE:

Les backups régulier n'ont pas d'auto-recover. Ils ne sont pas chargé automatiquement sur app02,
contrairement au backup dropbox qui fait un auto-recover toutes les nuits à 6h.


Pour faire un restore d'un backup régulier: (il faut se connecter en root)

	cd /home/fetch-slave
	./restore.sh ./tens/2015-07-19_12:00.gz


	Etapes de la manoeuvre de restore, telle que développée par 

	- copier le .gz à restaurer (pour éviter qu'il ne se fasse rm par la cron)
	- stopper les crons
	- stopper apache
	- stopper mySQL, attendre 5 secondes (va faire stopper une éventuelle cron en cours)
	- redémarrer mySQL ( pause ! 2 secondes )
	- restaurer le backup
	- si opération de test: sendComms doit être mis à zéro
	- restart apache
	- restart des crons

NOTES :

1. Il faut veiller à ce que le script lancé par la cron sur le SRV_PRODUCTION appartiennent à root, et soit exécutable par root

Si ce fichier est passé par exemple par FileZilla dans le dossier /home/petula
Il faut ensuite exécuter les commandes suivantes (noter les owner et permissions sur le fichier)

[root@app01 /]|PRODUCTION|# cd /srv/syst 
[root@app01 /srv/syst]|PRODUCTION|# mv /home/petula/dump-slave.sh .
[root@app01 /srv/syst]|PRODUCTION|# ls -l
total 8
-rwxr-xr-x 1 root   root   1316 Sep 19  2013 backup.sh
-rw-r--r-- 1 petula petula 1464 Nov 23 20:49 dump-slave.sh
[root@app01 /srv/syst]|PRODUCTION|# chown root dump-slave.sh
[root@app01 /srv/syst]|PRODUCTION|# chmod 755 dump-slave.sh
[root@app01 /srv/syst]|PRODUCTION|# ls -l
total 8
-rwxr-xr-x 1 root root   1316 Sep 19  2013 backup.sh
-rwxr-xr-x 1 root petula 1464 Nov 23 20:49 dump-slave.sh



2. Synchro du serveur slave:

La synchronisation de la DB sur le serveur mysql_slave nécéssite l'activation des logs mysql-bin.00XXXX.
Cette activation a lieu dans le fichier /etc/mysql/my.cnf

Les paramètres à préciser sont les suivant:

server-id               = 1
log_bin                 = /var/log/mysql/mysql-bin.log
expire_logs_days        = 4
max_binlog_size         = 100M
binlog_do_db            = mobminder


Voici le résultat concret produit dans le dossier /var/log/  ( commande ls -l introduite le 13 juin à 10h40 )
