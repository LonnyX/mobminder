Description incident grave et méthode de réparation : (pascal)

Le dimanche 7 juin 2020, la partition /tmp du serveur app04 était pleine (cause fichiers dropbox). 
La tâche cron weekly.php a échoué sur un OPTIMIZE TABLE archive_reservations (cette table fait 1.2Gb, le fichier temporaire n'a pas pu être écrit sur la partition tmp, la table archive_reservation s'est trouvée corrompue et le repair n'a pas fonctionnée). 
Sur le pc local dev, j'ai récupéré le backup du 6 juin sous format binaire (fichiers .MYI .MYD .frm). Leur unique présence dans un dossier du nom de la DB (mobminder) indique à MySQL l'existance de la DB et toutes les tables sont récupérées. Je procède de cette manière depuis 2011 sans soucis. Egalement depuis le passafe à MySQL 5.7, cette manière de faire fonctionne très bien. 
En local, j'ai fait un Dump SQL de la table archive_reservations non corrompue (puisque issue du backup). 
Le fichier archive_reservations.sql fait 3.2Gb (puisque c'est une version sous format d'instructions MySQL). Le fichier est en format utf8

Les tests suivant ont tous échoués (les caractères spéciaux sont tous corrompus éèàô... l'encodage utf8 n'est pas respécté):

	Voici comment examiner le défaut:
	- Dans l'application mobminder, en retournant 4 semaines en arrière, on voit tous les accents corrompus dans les notes de RDV
	- Dans Navicat, lorsqu'on ouvre la table, on voit tous les accents corrompus dans la colonne 'note'.


1- Transfert du fichier sur le serveur via FTP et exécution via 

mysql -u username -p mobminder < archive_reservations.sql

Cette référence indique que c'est une mauvaise méthode ( à cause de < ):
https://makandracards.com/makandra/595-dumping-and-importing-from-to-mysql-in-an-utf-8-safe-way


2- Transfert du même fichier via source

mysql> source /root/archive_reservations.sql

L'exécution d'un giga fichier SQL prend jusqu'à 1h30.


3- Transfert du même fichier au moyen de Navicat en utilisant la commande "Execute SQL file" depuis un click droit sur la DB mobminder.


4- Ecrire du code PHP qui tente de sélectionner les record avec une cnote corrompue, et y remplacer les caractères corrompus par le bon caractère. Je ne suis pas parvenu à distinguer les caractères corrompus... Il étaient tous pris pour de "é". 





Chaque test dure plus d'une heure. Afin de faire des tests plus rapides, j'ai décidé de sectionner le fichier en plusieurs sous parties.
J'ai utiliser la commande linux 
$ split --verbose -l600000 archive_reservations.sql arch_resa.
creating file 'arch_resa.aa'
creating file 'arch_resa.ab'
creating file 'arch_resa.ac'
creating file 'arch_resa.ad'
creating file 'arch_resa.ae'
...
creating file 'arch_resa.as' 

Une vingtaine de fichiers ont été créés.


J'ai effectué divers test jusqu'à celui-ci qui a fonctionné :

Chaque fichier a été converti du format utf8 vers utf8-bom

ET 

Dans le premier fichier (celui qui drop puis re-crèe la table), j'ai 

	1. commenté SET NAMES='utf8'
	2. changé archive_reservations en archive_test (pour faire un test avec seulement le 1er fichier et vérifier l'encodage via Navicat). 
	3. changé la ligne d'option de table en ENGINE = MyISAM CHARACTER SET = utf8 COLLATE = utf8_general_ci
	(voir entète du fichier arch_resa.aa.sql ci-dessous)

Ensuite j'ai renommé archive_test en archive_reservations et j'ai poursuivi avec les fichier b, c, d, e, ... s. 



NOTE:
depuis sa création, la web app écrit dans la DB sans utiliser SET NAMES='utf8'; Cette instruction devrait se trouver dans dbio.php et s'exécuter à chaque ouverture de connexion. Du fait qu'elle ne s'y trouve pas, les query du type LIKE "%bla%" ne respectent pas la collation utf8. C'est un problème à régler.

NOTE: 
Pour les essais et lors de la remise en place de la table, de nombreux test ont étés effectué et le disque /var/log/mysql qui contient les mysql-bin.007432 a été surchargé. Ceci a provoque un nouveau disk-full error. Je suis aller examiner la situation dans le dossier et j'ai supprimé les mysql-bin.006* pour faire de l'espace disque. Il faudra veiller à correctement dimensionner les partition lors du setup d'un prochain nouveau serveur.
